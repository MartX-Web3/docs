Echo's policy system is the trust boundary between you and your AI agent. It defines exactly what the agent is allowed to do — and the on-chain validator enforces those limits for every single UserOperation, regardless of what the gateway or agent says.

There are three layers, each building on the last.

---

## MetaPolicy (realtime mode)

MetaPolicy is always active. It governs every swap the agent executes in real time, whether or not a session is open.

| Field | Description |
|---|---|
| `dailyCap` | Maximum total spend (in USD equivalent) across all swaps in a rolling 24-hour window |
| `perTxMax` | Maximum amount for a single swap |
| `tokenWhitelist` | Array of ERC-20 addresses the agent is allowed to trade. Swaps involving any other token are rejected |
| `slippageCap` | Maximum acceptable slippage in basis points (e.g. `100` = 1%). UserOps specifying higher slippage are rejected |
| `paused` | Boolean kill switch. When `true`, all agent operations are rejected immediately. Can be set from the dashboard or by the agent calling `echo_pause_instance` |

MetaPolicy parameters are stored in your `PolicyInstance` on-chain and enforced by `EchoPolicyValidator` before any UserOp executes.

### Realtime signature mode

When operating under MetaPolicy alone (no active session), the gateway uses **realtime mode**:

```
Signature prefix: 0x03
Format:           [0x03][executeKey (32 bytes)]
```

The execute key is stored encrypted in the gateway keystore. It signs each UserOp individually. The on-chain validator verifies the signature against the registered execute key for your instance.

## SessionPolicy (autonomous mode)

A SessionPolicy is a scoped authorization that lets the agent execute a batch of operations **without per-op user confirmation**. It extends MetaPolicy — all MetaPolicy limits still apply, plus the session adds its own constraints.

Creating a session returns a `pendingTx` that you sign once via the dashboard. After that, the agent can execute ops autonomously until the session expires or the budget is exhausted.

| Field | Description |
|---|---|
| `tokenIn` | The input token for this session (single pair per session) |
| `tokenOut` | The output token for this session |
| `maxAmountPerOp` | Per-operation spend cap (cannot exceed MetaPolicy `perTxMax`) |
| `totalBudget` | Total spend allowed across the entire session lifetime |
| `maxOpsPerDay` | Maximum number of operations per calendar day within the session |
| `sessionExpiry` | Unix timestamp when the session automatically closes |

### Session signature mode

When executing within a session, the gateway uses **session mode**:

```
Signature prefix: 0x02
Format:           [0x02][sessionId (32 bytes)][sessionKey (32 bytes)]
```

The session key is generated by the gateway and registered on-chain when the session is created. It is stored encrypted in the keystore alongside the execute key. The on-chain validator checks the sessionId and session key signature — no execute key involvement needed.

### Example: creating a session via the agent

```
You: Set up a session to DCA 50 USDC into ETH over the next week, max 10 USDC per day.

Agent calls echo_create_session:
  tokenIn:       0xBa9D46448e4142AC7a678678eFf6882D9197d716  (USDC)
  tokenOut:      0xF0527287E6B7570BdaaDe7629C47D60a3e0eF104  (WETH)
  maxAmountPerOp: 10000000   (10 USDC)
  totalBudget:   50000000   (50 USDC)
  maxOpsPerDay:  1
  sessionExpiry: 1745280000  (7 days from now)

→ Returns pendingTx. Dashboard shows "Sign to activate session".
→ You approve in the dashboard.
→ Agent calls echo_execute_session once per day autonomously.
```

## Exploration Budget

The Exploration Budget is a small MetaPolicy allocation (typically 5–10 USDC) that lets the agent execute real swaps autonomously to **learn market conditions** — without requiring you to set up a full session first.

**Why it exists:**

Before recommending a large DCA session or a specific strategy, a capable agent wants to understand current prices, spreads, and gas costs. The Exploration Budget gives it permission to make a few small test swaps so it can calibrate recommendations with real data.

**How it works in practice:**

1. Your MetaPolicy includes a small `perTxMax` (e.g. 5 USDC) and `dailyCap` (e.g. 10 USDC) for exploratory use
2. The agent calls `echo_submit_intent` with small amounts to observe execution prices and slippage
3. Based on this data, the agent proposes a larger session (e.g. "I observed 0.08% slippage at this size — here is my recommended DCA plan")
4. You approve the session once, and the agent executes autonomously from there

<Tip>
  The Conservative template is well-suited as an Exploration Budget policy — its 20 USDC per-tx max and 100 USDC daily cap are large enough for meaningful exploration but small enough to limit downside if the agent misbehaves.
</Tip>

## Policy templates

When you create a new instance, you choose from three built-in templates. These are on-chain registered templates referenced by their template ID.

| Template | Template ID | Daily Cap | Per-Tx Max | Slippage Cap | Token Whitelist |
|---|---|---|---|---|---|
| **Conservative** | `0x153ccd...ea99` | 100 USDC | 20 USDC | 0.5% (50 bps) | USDC, WETH, DAI |
| **Standard** | `0x039a84...a971` | 1,000 USDC | 200 USDC | 1% (100 bps) | USDC, WETH, DAI, WBTC |
| **Active** | `0xb770bc...ac4ea` | 10,000 USDC | 2,000 USDC | 2% (200 bps) | USDC, WETH, DAI, WBTC, ARB, OP |

All templates start with `paused: false`.

## How to change your policy

**Via the dashboard:**
Navigate to Policy Settings and update any parameter. Changes are submitted as an on-chain transaction signed via WalletConnect (same one-shot QR pattern as onboarding).

**Via the agent:**
The agent can propose policy changes, but it cannot execute them unilaterally — policy updates always require a wallet signature from you. The agent will return a `pendingTx` for you to sign in the dashboard.

**Creating a new session:**
Call `echo_create_session` (via the agent or directly) to add a SessionPolicy on top of the current MetaPolicy. Sessions stack — you can have multiple active sessions for different token pairs simultaneously, each with independent budgets.
